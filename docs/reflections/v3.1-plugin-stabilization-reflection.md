## What v3.1 Actually Is

v3.0 was subtraction â€” cutting dead architecture until the nucleus was visible. v3.1 is the opposite: it's the **connection layer**. The MCP servers existed, the inference cycle existed, the governance kernel existed â€” but none of it was wired into a usable whole.

Three things were missing:
1. MCP servers didn't use real LLM reasoning â€” they stubbed out to kernel mocks
2. The plugin system compiled but the plugin file was silently excluded from the build
3. No marketplace or discovery â€” opencode couldn't find xray without manual config

v3.1 fixes all three.

## MCP Servers with Real LLM Reasoning

The biggest change is invisible from the diff stats. Before v3.1, `governance.server.ts` delegated to `handleGovernRequest` in the kernel â€” a thin pass-through that did basic validation but no real inference. The governance service was disconnected from the inference cycle.

Now the MCP governance server calls `GovernanceService.govern()` which runs a full inference cycle:

- Proposals are sent to real MCP skill servers (`code-review`, `security-audit`, `researcher`) that analyze them via LLM
- `VotingCoordinator` collects weighted votes per agent expertise
- The inference cycle persists state (`inference-cycle-state.json`)
- External Dynamo Solar SSOT is a hard requirement (no local mode fallback)

Seven MCP server tool endpoints now do LLM-powered analysis instead of mock responses. The governance pipeline is real end-to-end.

## Plugin System: The Secret Bug

The single most embarrassing fix in this release: `xray-codex-injection.ts` was **excluded from tsconfig.json** (`src/plugin/xray-codex-injection.ts` in the exclude list). The build script had a fallback:

```sh
find dist/plugin -name '*codex-injection*.js' -exec cp {} dist/plugin/xray-codex-injection.js \; 2>/dev/null || true
```

This never found anything because nothing created the file. The `|| true` silently swallowed the failure. The plugin existed in source, existed in `package.json` as `opencode.plugin`, but never shipped to `dist/`.

The postinstall and E2E tests guarded against this with `HAS_DIST` checks, which meant they silently skipped the plugin tests in CI. Green suite, broken artifact.

Fixes applied:

- Removed the exclude from `tsconfig.json` â€” plugin now compiles with the rest of the project
- Added `ConfigLoader.clearConfigPaths()` for test isolation (tests were picking up the real `.mcp.json`)
- Updated test assertions to match reality (`.mcp.json` exists now, version is 3.1.1)
- Reduced governance MCP remote test timeout from 4s to 2s

## Marketplace Discovery

- `.grok-plugin/plugin.json` â€” xAI Grok marketplace entry

The `ConfigLoader` was updated to handle the `mcpServers` object format (vs the old array format), making it compatible with both conventions.

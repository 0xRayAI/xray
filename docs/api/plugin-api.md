# Plugin API (v3 — Frozen)

Third-party plugin authors can extend xray in two ways:

1. **SkillPlugin** — Register a governance skill that analyzes proposals (Code Review, Security Audit, Researcher style)
2. **MetamorphosisEngine** — Hook into the self-evolution lifecycle to observe or generate proposals

---

## 1. SkillPlugin Interface (Governance Skills)

Import from the nucleus barrel:

```ts
import {
  pluginRegistry,
  type SkillPlugin,
  type SkillProposalArgs,
  type SkillPluginResult,
} from '0xray';
```

### Interface Definitions

```ts
export interface SkillPlugin {
  /** Unique skill name (e.g. "my-custom-analyzer"). Must not collide with built-in skills. */
  name: string;

  /** Called with each proposal requiring skill analysis. Must return structured content. */
  analyzeProposal(args: SkillProposalArgs): Promise<SkillPluginResult>;
}

export interface SkillProposalArgs {
  proposalTitle?: string;
  proposalDescription?: string;
  evidence?: string[];
  proposalType?: string;
}

export interface SkillPluginResult {
  content: Array<{ type: string; text: string }>;
}
```

### Registration

```ts
import { pluginRegistry } from '0xray';

pluginRegistry.register({
  name: 'my-skill',
  async analyzeProposal(args) {
    // Analyze the proposal and return structured content
    return {
      content: [{ type: 'text', text: 'Results of my analysis' }],
    };
  },
});
```

### Resolution Priority

| Priority | Source |
|----------|--------|
| 1 (first) | Dynamically registered via `pluginRegistry.register()` (for custom/non-built-in names only) |
| 2 (fallback) | MCP governance server (stdio / Streamable HTTP) |
| 3 (last) | In-process built-in (`code-review`, `security-audit`, `researcher`) |

**Note**: Built-in skill names are reserved. The `isBuiltInSkill` guard in `GovernanceService` ensures built-ins always route to the standard MCP/in-process implementation and cannot be overridden by dynamic plugins. This matches the "Built-in names are reserved" restriction.

### Restrictions
- Built-in names are reserved: `code-review`, `security-audit`, `researcher`
- Name must match `^[a-z][a-z0-9-]{2,63}$`
- `analyzeProposal` must resolve within 30s or the call is timed out by governance

## 2. MetamorphosisEngine Interface (Self-Evolution Lifecycle)

Import from the nucleus barrel:

```ts
import {
  type MetamorphosisEngine,
  type MetamorphosisProposal,
} from '0xray';
```

### Interface Definitions

```ts
export interface MetamorphosisEngine {
  /** Human-readable engine name (for logging). Must be unique across all engines. */
  name: string;

  /** Called when PostProcessor enters a lifecycle phase.
   *  Known phases: 'monitoring-complete', 'post-process-complete'
   *  Future phases may be added — engines MUST handle unknown phases gracefully. */
  onPhase?(phase: string, context: unknown): Promise<void>;

  /** Called when a metamorphosis proposal is generated by any engine.
   *  This is a broadcast — all registered engines receive it. */
  onProposal?(proposal: MetamorphosisProposal): Promise<void>;
}

export interface MetamorphosisProposal {
  id: string;
  type: 'add' | 'modify' | 'remove';
  target: string;
  description: string;
  rationale: string;
  impact: 'low' | 'medium' | 'high';
}
```

### Constructor Injection

Engines are provided to PostProcessor at construction time:

```ts
import { PostProcessor, SelfProposalEngine } from '0xray';

const pp = new PostProcessor(
  stateManager,
  sessionMonitor,
  config,
  [
    new SelfProposalEngine({ /* config */ }),
    myCustomEngine,
  ],
);
```

### Known Lifecycle Phases

| Phase | When | Use Case |
|-------|------|----------|
| `monitoring-complete` | CI/CD monitoring finished with success | Detect patterns, submit self-proposals |
| `post-process-complete` | Full post-processor loop completed | Post-session analysis, cleanup |

Engines must not throw on unknown phases; unknown phases are silently ignored.

## 3. SelfProposalConfig

Built-in engine that implements MetamorphosisEngine. First-party only but config is part of the public API:

```ts
export interface SelfProposalConfig {
  maxProposalsPerHour?: number;          // default: 1
  circuitBreakerThreshold?: number;      // default: 3
  circuitBreakerCooldownHours?: number;  // default: 24
  activityLogPath?: string;              // default: 'logs/framework/activity.log'
  whitelistedTargets?: string[];         // default: ['config/', 'features.json', 'src/processors/']
  maxLogLines?: number;                  // default: 100
  errorThreshold?: number;               // default: 5
  warningThreshold?: number;             // default: 10
  governanceRejectionThreshold?: number; // default: 3
  proposalConfidence?: number;           // default: 0.7
  metamorphosisThreshold?: number;       // default: 0.7
}
```

## 5. Stability Guarantee

These interfaces are **frozen** as of v3.0.0:

- Signature changes will be additive only (new optional fields / methods)
- Deprecation will follow a 3-minor-version cycle before removal
- Any breaking change will be documented in `docs/migration/` and announced in CHANGELOG

The following are **NOT** covered by the stability guarantee:
- Internal implementation classes (e.g., `PluginRegistryImpl`, `SelfProposalEngine` internal methods)
- Interfaces in `src/integrations/plugins/plugin-registry.ts` (the industrial-grade registry is internal)
- Any type prefixed with `Internal` or marked `@internal` in JSDoc
